เอกสารเปรียบเทียบ Feature Engineering (Version 2)

เป้าหมายของ Version 2: ลดความซ้ำซ้อนและมิติของข้อมูล (Dimensionality Reduction) จาก 136 คอลัมน์ ให้เหลือ 38 คอลัมน์ โดยใช้วิธี Target Encoding เพื่อดึงค่าสถิติความเสี่ยงมาใช้แทน One-Hot Encoding รวมถึงการสร้าง Smart Features ใหม่ๆ เพื่อเพิ่มประสิทธิภาพการเรียนรู้ของโมเดล

1. ตัวแปรที่ถูกตัดทิ้ง (Dropped Features)

กลุ่มข้อมูลเหล่านี้ถูกคัดกรองออกเนื่องจาก 3 เหตุผลหลัก คือ 1) ข้อมูลระบุตัวตนที่ไม่ช่วยในการทำนาย 2) ข้อมูลหลังจากการสั่งซื้อที่ก่อให้เกิด Data Leakage และ 3) ข้อมูลที่เกี่ยวกับระบบเก่า

ชื่อ Feature เหตุผลที่ตัดทิ้ง (Drop Logic)
order_id, customer_id, product_id, supplier_id, courier_id เป็นรหัส ID (Identifier) ที่เป็นตัวเลขสุ่ม ไม่มีความหมายเชิงสถิติ
customer_name, customer_phone, product_name, supplier_name, supplier_contact ข้อมูลส่วนบุคคลและข้อมูลตัวหนังสือ (Text) ที่ไม่จำเป็นต่อโครงสร้างโมเดล
expected_delivery_date, delivery_date วันเวลาในอนาคต ทำให้เกิดข้อมูลรั่วไหล (Data Leakage) เพราะตอนที่ลูกค้ากดสั่งซื้อ เรายังไม่รู้วันส่งของจริง
return_id, return_date, return_reason, return_scenario, item_condition, return_status, refund_amount ข้อมูลที่เป็นผลลัพธ์ของการคืนสินค้า (Leakage) ซึ่งจะรู้ก็ต่อเมื่อลูกค้าคืนของแล้วเท่านั้น
score_id, risk_score, risk_tier, scored_at, shap_values ข้อมูลการทำนายจากระบบเก่า (System Artifacts) ไม่สามารถนำมาตั้งต้นทำนายใหม่ได้

2. ตัวแปรที่ถูกสร้างขึ้นมาใหม่ (Engineered Smart Features)

การสกัดข้อมูล (Feature Extraction) เพื่อให้ตัวแปรสื่อความหมายทางธุรกิจมากขึ้น

ชื่อ Feature ใหม่ ที่มาและประโยชน์ทางธุรกิจ
customer_tenure_months คำนวณจาก (วันสั่งซื้อ - วันสมัครสมาชิก) เพื่อดูว่า "อายุการเป็นสมาชิก" ส่งผลต่อการทิ้งของหรือไม่ (ลูกค้าใหม่มักทิ้งของมากกว่าลูกค้าเก่า)
order_month, order_dayofweek, is_weekend สกัดออกมาจาก order_date เพื่อดูความสัมพันธ์ของการสั่งซื้อในวันหยุด หรือช่วงปลายเดือน
age_group จัดกลุ่มอายุลูกค้า (Binning) ออกเป็นช่วง เช่น <20, 20-30, 30-40 เพื่อให้โมเดลจับรูปแบบพฤติกรรมคนแต่ละเจเนอเรชันได้ง่ายกว่าการใช้อายุแบบตัวเลขเดี่ยวๆ
logistics_risk สูตร: damage_rate * is_fragile นำอัตราการทำของพังของขนส่ง มาคูณกับ ความเปราะบางของสินค้า เป็นดัชนีชี้วัดความเสี่ยงพัสดุพัง

3. การแปลงตัวแปรหมวดหมู่ด้วย Target Encoding (ความลับที่ทำให้ V2 แม่นขึ้น)

แทนที่จะใช้ One-Hot Encoding ซึ่งสร้างคอลัมน์ขยะจำนวนมหาศาล V2 เปลี่ยนมาใช้ Target Encoding (Mean Encoding) คือการเปลี่ยน "ชื่อหมวดหมู่" ให้เป็น "เปอร์เซ็นต์อัตราการคืนของของหมวดหมู่นั้นๆ" โดยตรง พร้อมใส่สมการ Smoothing ป้องกัน Overfitting

ตัวแปรที่ใช้ Target Encode กลไกการทำงาน
province เปลี่ยนชื่อจังหวัด เช่น "กรุงเทพ" เป็น อัตราตีกลับเฉลี่ยของคนกทม. (เช่น 0.22) โมเดลจะรู้ทันทีว่าจังหวัดนี้เสี่ยงน้อยหรือมาก
brand, category เปลี่ยนเป็นค่าความเสี่ยงของสินค้านั้นๆ (เช่น เสื้อผ้ามักโดนตีกลับสูงกว่าเครื่องใช้ไฟฟ้า)
payment_method, channel_type, courier_name, courier_type แปลงวิธีจ่ายเงิน ขนส่ง และช่องทาง ให้เป็นตัวเลขความเสี่ยง
gender, membership_tier, promo_name, promo_type, age_group แปลงหมวดหมู่ข้อมูลลูกค้าและโปรโมชั่นให้เป็นคะแนนความเสี่ยงเชิงพฤติกรรม

4. ตัวแปรที่เก็บไว้ใช้งานตามเดิม (Kept Features)

ตัวแปรประเภทตัวเลข (Numeric) ที่สามารถนำมาใช้งานได้เลย จะถูกเก็บไว้และนำไปทำ Scaling (ปรับสเกล) เพื่อป้อนเข้าโมเดล

รายการตัวแปรตัวเลขที่ถูกเก็บรักษาไว้
quantity, unit_price, tier_discount_pct, campaign_discount_pct, total_discount_pct, discount_applied_amount, total_amount (กลุ่มข้อมูลการเงิน/จำนวน)
delivery_time_expected_days, delivery_days, delay_days (กลุ่มข้อมูลประเมินระยะเวลาจัดส่ง)
is_repurchased_item, order_hour, days_since_last_order, hist_order_count, hist_return_rate (กลุ่มข้อมูลประวัติพฤติกรรมและ Interaction)
age (แม้นำไปทำอายุกลุ่มแล้ว แต่ยังคงเก็บตัวเลขอายุไว้ด้วย), damage_rate

สรุปผลกระทบต่อสถาปัตยกรรมข้อมูล (Data Architecture Impact)

จากการตัดตัวแปรที่ไม่ได้ใช้ (Drop) สร้างตัวแปรใหม่ (Engineer) และยุบรวมด้วย Target Encoding ทำให้จำนวนมิติข้อมูล (Dimensionality) เปลี่ยนแปลงดังนี้:

  • Version 1: 136 คอลัมน์ (มีสัญญาณรบกวนเยอะ โมเดลงง)
  • Version 2: 38 คอลัมน์ (มีแต่ตัวแปรเชิงสถิติที่เข้มข้น โมเดลเรียนรู้ได้แม่นยำขึ้นมาก)
